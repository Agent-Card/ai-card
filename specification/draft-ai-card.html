1. Introduction

The AI ecosystem comprises a growing number of protocols, artifact formats, and service types. Model Context Protocol (MCP) servers, Agent-to-Agent (A2A) agents, Anthropic skills, datasets, model cards, and other AI artifacts each define their own metadata and discovery mechanisms. This fragmentation forces clients and registries to implement bespoke logic for each artifact type, increasing complexity and reducing interoperability.¶

This document defines two complementary standards:¶

AI Catalog
A typed, nestable container for discovering heterogeneous AI artifacts. Each entry declares its artifact type via a media type and may reference or inline the native artifact metadata. Catalogs themselves may be nested as entries, enabling composable bundles analogous to plugin packages.¶
Trust Card (AI Card)
A portable, verifiable identity and trust document that accompanies any AI artifact as a peer element. The Trust Card functions as a "passport" for an AI artifact: it proves the artifact's identity and carries attestations, provenance, and compliance metadata without wrapping or modifying the artifact's native format.¶

The AI Catalog is intentionally agnostic about the artifacts it indexes. It does not define or constrain the schema of MCP server manifests, A2A agent cards, or any other artifact format. Instead, it relies on media types to identify what each entry is, and delegates the definition of artifact-specific metadata to the respective protocol specifications.¶

1.1. Conventions and Definitions

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in BCP 14 [RFC2119] [RFC8174] when, and only when, they appear in all capitals, as shown here.¶

The following terms are used throughout this document:¶

AI Catalog
A JSON document conforming to the application/ai-catalog+json media type that contains an ordered list of catalog entries.¶
Catalog Entry
A single item in an AI Catalog, identified by a media type and referencing or inlining an AI artifact.¶
Trust Card (AI Card)
A JSON object providing verifiable identity, attestation, and provenance metadata for an AI artifact.¶
Artifact
Any AI resource described by a catalog entry, such as an MCP server manifest, an A2A agent card, an Anthropic skill definition, a dataset descriptor, or a nested AI Catalog.¶

2. Design Goals

  1. Artifact Agnosticism: The catalog MUST be capable of indexing any type of AI artifact without requiring knowledge of the artifact's internal schema.¶

  2. Media Type Identification: Each catalog entry MUST declare its artifact type using a media type, enabling clients to select, filter, and route entries without parsing artifact content.¶

  3. Composability: Catalogs MUST support nesting, allowing a catalog entry to reference another AI Catalog. This enables plugin-style packaging where a single catalog bundles multiple related artifacts.¶

  4. Trust as a Peer: Identity and trust metadata MUST NOT wrap or modify the native artifact. Instead, a Trust Card accompanies an entry as a peer element, keeping artifact formats pristine.¶

  5. Static Discovery: The standard MUST define a well-known URL for serving AI Catalogs, enabling crawlers and clients to discover available artifacts for a given domain.¶

3. AI Catalog

3.1. Media Type

An AI Catalog document is identified by the media type:¶

application/ai-catalog+json
¶

This media type is registered with IANA as specified in Section 7.2.¶

3.2. Top-Level Structure

An AI Catalog document is a JSON object that MUST contain the following members:¶

$schema
A string containing a URI [RFC3986] to the JSON Schema for this catalog format.¶
specVersion
A string indicating the version of this specification that the catalog conforms to, in "Major.Minor" format (e.g., "1.0").¶
entries
An array of Catalog Entry objects as defined in Section 3.4. This array MAY be empty.¶

The following members are OPTIONAL:¶

host
A Host Info object as defined in Section 3.3 identifying the operator of this catalog.¶
metadata
An open map of string keys to arbitrary values for custom or non-standard metadata.¶

3.3. Host Info

The Host Info object identifies the operator of the catalog. It MUST contain:¶

name
A string containing the human-readable name of the host (e.g., the organization name).¶

The following members are OPTIONAL:¶

id
A string containing a verifiable identifier for the host (e.g., a DID or domain name).¶
documentationUrl
A string containing a URL to the host's documentation.¶
logoUrl
A string containing a URL to the host's logo.¶
trustCard
A Trust Card object as defined in Section 4 providing verifiable identity and trust metadata for the host itself.¶

3.4. Catalog Entry

A Catalog Entry object describes a single AI artifact in the catalog. It MUST contain the following members:¶

id
A string containing a globally unique identifier for this entry. This SHOULD be a URN [RFC8141] or URI [RFC3986].¶
name
A string containing a human-readable name for the artifact.¶
mediaType

A string containing the media type that identifies the type of the referenced artifact. This is the mechanism by which clients determine what kind of AI artifact the entry represents. Well-known values include (but are not limited to):¶

  • application/ai-catalog+json — a nested AI Catalog¶
  • application/a2a-agent+json — an A2A Agent Card¶
  • application/mcp-server+json — an MCP Server manifest¶
  • Other media types as defined by respective protocol specifications¶

A Catalog Entry MUST contain exactly one of the following members to provide the artifact content:¶

url
A string containing a URL where the full artifact document can be retrieved. The document served at this URL SHOULD be served with the media type declared in the mediaType field.¶
inline
A JSON value containing the complete artifact document inline. The structure of this value is determined by the mediaType field and is opaque to this specification.¶

The following members are OPTIONAL:¶

description
A string containing a short description of the artifact.¶
tags
An array of strings serving as keywords for filtering and discovery.¶
publisher
A Publisher object as defined in Section 3.5 identifying the entity that publishes this artifact. This is the sole location for publisher information; it is not duplicated in the Trust Card.¶
trustCard
A Trust Card object as defined in Section 4 providing verifiable identity and trust metadata for this specific artifact. This is the "passport" that travels alongside the artifact as a peer element.¶
updatedAt
A string containing an ISO 8601 [RFC3339] timestamp indicating when this entry was last modified.¶
metadata
An open map of string keys to arbitrary values for custom data.¶

3.4.1. Nested Catalogs

When the mediaType of a Catalog Entry is application/ai-catalog+json, the entry represents a nested AI Catalog. The nested catalog MAY be provided via the url field (referencing an external catalog document) or via the inline field (embedding the full catalog).¶

Nested catalogs enable composable packaging. For example, a vendor may publish a single catalog entry that bundles an A2A agent, an MCP server, and a dataset as a plugin package:¶

Vendor Catalog (application/ai-catalog+json)
  └── Entry: "Finance Plugin" (application/ai-catalog+json)
        ├── Entry: Finance A2A Agent (application/a2a-agent+json)
        ├── Entry: Finance MCP Server (application/mcp-server+json)
        └── Entry: Market Dataset (application/parquet)
¶

Clients processing nested catalogs SHOULD impose a maximum nesting depth to prevent circular references. A depth limit of 8 is RECOMMENDED.¶

3.5. Publisher Object

The Publisher object identifies the entity responsible for an artifact. It appears on the Catalog Entry and is the canonical location for publisher information. It MUST contain:¶

id
A string containing a verifiable identifier for the publisher organization.¶
name
A string containing the human-readable name of the publisher.¶

The following members are OPTIONAL:¶

identityType
A string providing a type hint for the publisher identifier (e.g., "did", "dns").¶

4. Trust Card (AI Card)

The Trust Card is a JSON object that provides verifiable identity and trust metadata for any AI artifact. It functions as a "passport": proving the artifact's identity and carrying attestations, provenance, and compliance metadata.¶

The Trust Card does NOT wrap the artifact. It sits alongside the artifact as a peer element within a Catalog Entry, keeping the native artifact format unmodified.¶

Publisher information is NOT duplicated in the Trust Card. The informational publisher identity is carried on the Catalog Entry (see Section 3.4). The Trust Card can verify publisher identity through an attestation of type "publisher-identity" in its attestations array.¶

4.1. Identity Binding

A Trust Card object MUST contain:¶

identity
A string containing a globally unique URI [RFC3986] that serves as the primary subject identifier for this artifact. This SHOULD be a DID, SPIFFE ID, or URL.¶

When a Trust Card appears within a Catalog Entry, the Trust Card's identity field MUST match the entry's id field. This binding ensures the trust claims are unambiguously associated with the catalog artifact. Consumers MUST reject a Trust Card whose identity does not match the containing entry's id.¶

When a Trust Card appears on a Host Info object, the Trust Card's identity SHOULD match the host's id field when present.¶

4.2. Optional Members

The following members are OPTIONAL:¶

identityType
A string providing a type hint for the identity URI (e.g., "did", "spiffe", "dns"). This field is OPTIONAL when the type is evident from the URI scheme.¶
trustSchema
A Trust Schema object as defined in Section 4.3.¶
attestations
An array of Attestation objects as defined in Section 4.4. This is the mechanism for verifiable claims including publisher identity verification (using attestation type "publisher-identity"), compliance certifications, and other proofs.¶
provenance
An array of Provenance Link objects as defined in Section 4.5.¶
privacyPolicyUrl
A string containing a URL to the privacy policy governing this artifact.¶
termsOfServiceUrl
A string containing a URL to the terms of service.¶
signature
A string containing a detached JWS [RFC7515] signature computed over the Trust Card content. This enables integrity verification of the trust metadata independent of the artifact.¶
metadata
An open map of string keys to arbitrary values for extending trust metadata.¶

4.3. Trust Schema Object

A Trust Schema object describes the trust framework applied to the artifact. It MUST contain:¶

id
A string identifying the trust schema.¶
version
A string indicating the schema version.¶

The following members are OPTIONAL:¶

governanceUri
A string containing a URI to the governance policy document.¶
verificationMethods
An array of strings identifying the verification methods supported (e.g., "did", "x509", "dns-01").¶

4.4. Attestation Object

An Attestation object provides verifiable proof of a claim. It MUST contain:¶

type
A string identifying the attestation type (e.g., "SOC2-Type2", "HIPAA-Audit", "ISO27001").¶
uri
A string containing the location of the attestation document. This may be an HTTPS URL or an inline Data URI [RFC2397].¶
mediaType
A string indicating the format (e.g., "application/pdf", "application/jwt").¶

The following members are OPTIONAL:¶

digest
A string containing a cryptographic hash for integrity verification (e.g., "sha256:abcd1234...").¶
size
An unsigned integer indicating the size of the attestation in bytes.¶
description
A string containing a human-readable label.¶

4.5. Provenance Link Object

A Provenance Link records lineage information. It MUST contain:¶

relation
A string describing the relationship (e.g., "materializedFrom", "derivedFrom", "publishedFrom").¶
sourceId
A string identifying the source artifact or data.¶

The following members are OPTIONAL:¶

sourceDigest
A string containing the digest of the source.¶
registryUri
A string containing the URI of the registry holding the source.¶
statementUri
A string containing the URI of a provenance statement document.¶
signatureRef
A string referencing the key used to sign the provenance statement.¶

5. Static Discovery

5.1. Well-Known URI

Hosts that wish to advertise AI artifacts MUST serve an AI Catalog document at the following well-known URI [RFC8615]:¶

/.well-known/ai-catalog.json
¶

The document MUST be served with the media type application/ai-catalog+json.¶

Clients discovering AI artifacts for a given domain SHOULD attempt to retrieve this well-known URL. If a valid AI Catalog document is returned, the client SHOULD use the url entries to retrieve individual artifacts and their associated Trust Cards.¶

5.2. Dynamic Discovery

Implementing protocols MAY support dynamic catalog generation through their own mechanisms, such as providing different catalog content based on a caller's identity or query parameters. Defining dynamic discovery behavior is out of scope for this specification.¶

6. Security Considerations

6.1. Trust Card Integrity

Publishers SHOULD sign Trust Cards using a detached JWS [RFC7515] in the signature field. Consumers SHOULD verify the signature before trusting the identity and attestation claims in the Trust Card.¶

6.2. Transport Security

AI Catalogs, artifacts, and Trust Cards MUST be served over HTTPS (TLS 1.2 or later) to prevent tampering and eavesdropping.¶

6.3. Nested Catalog Depth

Clients processing nested catalogs MUST enforce a maximum recursion depth to prevent denial-of-service attacks via deeply nested or circular catalog references. A maximum depth of 8 is RECOMMENDED.¶

6.4. Publisher Identity Verification

Consumers SHOULD verify publisher identity through the verification methods described in the Trust Card's trust schema before granting elevated trust to an artifact.¶

6.5. Privacy Considerations

Logo URLs SHOULD use Data URIs [RFC2397] to avoid leaking client information through image fetch requests. Publishers SHOULD carefully consider what information is included in metadata extension fields.¶

Appendix A. CDDL Schema

The following CDDL [RFC8610] defines the normative schema for AI Catalog and Trust Card documents.¶

A.1. AI Catalog 

AICatalog = {
  $schema: text,
  specVersion: text,
  ? host: HostInfo,
  entries: [* CatalogEntry],
  ? metadata: { * text => any }
}

HostInfo = {
  name: text,
  ? id: text,
  ? documentationUrl: text,
  ? logoUrl: text,
  ? trustCard: TrustCard
}

CatalogEntry = {
  id: text,
  name: text,
  mediaType: text,
  (url: text // inline: any),
  ? description: text,
  ? tags: [* text],
  ? publisher: Publisher,
  ? trustCard: TrustCard,
  ? updatedAt: tdate,
  ? metadata: { * text => any }
}
¶

A.2. Trust Card (AI Card) 

TrustCard = {
  identity: text,
  ? identityType: text,
  ? trustSchema: TrustSchema,
  ? attestations: [* Attestation],
  ? provenance: [* ProvenanceLink],
  ? privacyPolicyUrl: text,
  ? termsOfServiceUrl: text,
  ? signature: text,
  ? metadata: { * text => any }
}

Publisher = {
  id: text,
  name: text,
  ? identityType: text
}

TrustSchema = {
  id: text,
  version: text,
  ? governanceUri: text,
  ? verificationMethods: [* text]
}

Attestation = {
  type: text,
  uri: text,
  mediaType: text,
  ? digest: text,
  ? size: uint,
  ? description: text
}

ProvenanceLink = {
  relation: text,
  sourceId: text,
  ? sourceDigest: text,
  ? registryUri: text,
  ? statementUri: text,
  ? signatureRef: text
}

Appendix B. Example: Multi-Artifact Catalog with Nested Plugin

{
  "$schema": "https://agent-card.github.io/ai-catalog/schema.json",
  "specVersion": "1.0",
  "host": {
    "name": "Acme Services Inc.",
    "id": "did:example:org-acme-corp",
    "documentationUrl": "https://docs.acme-corp.com/ai"
  },
  "entries": [
    {
      "id": "urn:acme:agent:finance-a2a",
      "name": "Acme Finance A2A Agent",
      "mediaType": "application/a2a-agent+json",
      "url": "https://api.acme-corp.com/agents/finance.json",
      "description": "A2A agent for financial workflows.",
      "tags": ["finance", "a2a"],
      "publisher": {
        "id": "did:example:org-acme",
        "name": "Acme Financial Corp"
      },
      "trustCard": {
        "identity": "urn:acme:agent:finance-a2a",
        "identityType": "did",
        "attestations": [
          {
            "type": "publisher-identity",
            "uri": "https://trust.acme.com/certs/publisher.jwt",
            "mediaType": "application/jwt",
            "description": "Verifies did:example:org-acme as publisher"
          },
          {
            "type": "SOC2-Type2",
            "uri": "https://trust.acme.com/reports/soc2.pdf",
            "mediaType": "application/pdf",
            "digest": "sha256:a1b2c3d4e5f6"
          }
        ],
        "privacyPolicyUrl": "https://acme.com/legal/privacy",
        "termsOfServiceUrl": "https://acme.com/legal/terms"
      },
      "updatedAt": "2026-03-15T10:00:00Z"
    },
    {
      "id": "urn:acme:server:finance-mcp",
      "name": "Acme Finance MCP Server",
      "mediaType": "application/mcp-server+json",
      "url": "https://api.acme-corp.com/mcp/finance.json",
      "description": "MCP server with finance tools.",
      "tags": ["finance", "mcp"],

    {
      "id": "urn:acme:plugin:finance-suite",

            "id": "urn:acme:server:finance-mcp",

          },
          {
            "id": "urn:acme:data:market-2026q1",
            "name": "Market Dataset Q1 2026",
            "mediaType": "application/parquet",
            "url": "https://data.acme-corp.com/market-2026q1.parquet",
            "trustCard": {
              "identity": "urn:acme:data:market-2026q1",

                {
                  "relation": "publishedFrom",

Appendix D. Trust Card as a Passport — Conceptual Model

The Trust Card is designed as a peer to the artifact, not a wrapper. This keeps the native artifact format pristine while still enabling rich trust and identity verification.¶

┌─────────────────────────────────────────────────┐
│              Catalog Entry                      │
│                                                 │
│  ┌─────────────────┐   ┌─────────────────────┐ │
│  │   AI Artifact    │   │   Trust Card         │ │
│  │   (native fmt)   │   │   "Passport"         │ │
│  │                  │   │                      │ │
│  │  MCP Server      │   │  identity            │ │
│  │  A2A Agent       │   │  publisher (verified)│ │
│  │  Anthropic Skill │   │  attestations        │ │
│  │  Dataset          │   │  provenance          │ │
│  │  Nested Catalog  │   │  signature           │ │
│  └─────────────────┘   └─────────────────────┘ │
│         url/inline             trustCard         │
└─────────────────────────────────────────────────┘
¶

The artifact and its passport are independent documents that can be verified separately. A client can fetch the artifact without examining trust metadata, or verify trust metadata without fetching the full artifact.¶
